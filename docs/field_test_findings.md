## 3. 効いている点（構造が違っても共通して機能）

- **主要テキスト項目の入力**: 会社名・氏名・メール・問い合わせ本文（署名付き）・個人情報同意チェックを、全く異なる構造の実フォームで正しく入力
- **ハニーポット検知・非入力**: honeypot判定した隠しフィールドを埋めない（company #9で実際に email ロールのhoneypot欄をスキップ）
- **reCAPTCHA v2/v3 検知**: 検知したら gate=block ＝ 手動送信キューへ回す
- **確認画面ハンドリング**: 「確認画面へ」ボタンを分類してクリックし、確認画面/入力済みフォームをスクショ
- **コンプラ抑制**: 「営業お断り」系文言を検知して自動除外
- **送信の非実行**: Planフェーズで最終送信を一切しない設計が実サイトでも守られた

## 4. 発見した課題（改善が必要）

スクショ（[artifacts/plan_*.png](../artifacts/)）で露見した、複雑フォームで未入力が残る要因。

### 課題A（最優先）— 分割フィールド未対応
1つの論理項目が複数の `<input>` に分かれているケースを、現状は先頭のボックスに全部入れてしまう。
- **電話番号** 3分割（市外/市内/番号）: 例 ダイセル物流で「03-1」だけ入り検証エラー
- **郵便番号** 2分割
- **氏名** 姓/名 分割: 「福井 翔」を姓欄に丸ごと投入
- **住所** 都道府県(select)＋市区町村(text) 分割

**影響度**: 大。日本のフォームの多くが電話・氏名・住所を分割。これが未対応だと mid/low に留まる主因。

### 課題B — フリガナ（kana）未生成
氏名のフリガナが必須の場合でも生成していない（送信者名「福井 翔」が漢字のため空のまま）。→ 必須バリデーションで弾かれる。

### 課題C — 必須の select / radio 未選択
- お問い合わせ種別・ご希望の商談種別などの **必須プルダウン** を自動選択していない
- 介護状況・認知症などの **必須ラジオ**（※これは消費者向けフォームで、そもそも対象外の可能性）

### 課題D — メール確認欄（confirm email）未対応
「メールアドレス（確認）」の再入力欄にロールが割り当たらず空になる。

### 課題E — フォームの「種類」の見分け
介護相談・資料請求(brochure)など、**B2B営業窓口でない消費者向けフォーム**を発見してしまうことがある（例 #2 ミモザ＝介護相談、#9 みんかい＝/brochure 資料請求, conf=0.223）。営業文面と項目がミスマッチ。

### 課題F — 発見率（大手のiframe/SPA）
戸田建設・前田建設・JR貨物・ツクイ 等は静的にもブラウザレンダリングでもフォーム未検出。**クロスオリジンiframe埋込**（HubSpot/Marketo等）や独自SPAが原因。

## 5. 改善案（優先度順）

| 優先 | 課題 | 改善案 | 想定効果 |
|---|---|---|---|
| **P1** | A 分割フィールド | L2で「同一ラベル配下の連続input群」を検出し、電話/郵便/氏名/住所を**サブフィールド分割ルール**で個別マッピング。値も分割して投入（電話→ハイフン分割、氏名→姓/名） | mid/lowの多くがhighへ。未入力の最大要因を解消 |
| **P1** | B フリガナ | 送信者情報に `SENDER_KANA_SEI/MEI` を追加し、kanaロールに投入。将来はLLMで漢字→カナ推定も可 | フリガナ必須フォームが通る |
| **P2** | C 必須select/radio | 必須の `<select>` は「選択してください」以外の先頭値 or ラベル一致で自動選択。ラジオは「その他/法人/営業」系を優先選択。**曖昧なものはgateを下げて人間へ** | 確認画面到達率↑ |
| **P2** | D メール確認欄 | ラベルに「確認/再入力/confirm」を含むメール型は `email_confirm` ロールとして email と同値を投入 | 確認欄ありフォームが通る |
| **P2** | E フォーム種別判定 | ページ/フォーム文言から「相談・資料請求・予約・採用応募」等の**非営業窓口を分類**し、gate=lowまたは除外候補に。真のB2B問い合わせ窓口を優先 | 誤フォーム送信の防止（ブランド保護） |
| **P3** | F iframe/SPA発見 | 同一オリジンiframe内フォームの検出・入力対応。クロスオリジン(HubSpot等)は埋込元URLを辿るか手動キューへ | 大手の発見率↑（ただし実ICPでは優先度低） |

### 実装順の推奨
1. **課題A（分割フィールド）** から着手 ── 費用対効果が最大。L2のマッピングとL4の値投入の両方に手を入れる
2. 次に **B（フリガナ）＋D（メール確認）** ── 設定追加＋ロール追加で軽量
3. その後 **C（必須select/radio）** と **E（フォーム種別判定）** ── 品質・ブランド保護の底上げ
4. **F（iframe/SPA）** は P2フェーズで（実ICPのSMBで発見率を実測してから判断）

## 6. 再現方法

```bash
cd toiawase_form_sales_auto_system
npm run cli -- ingest scripts/test20_companies.csv
npm run cli -- discover --limit 30      # L1+L2（実ブラウザ、数分）
npm run cli -- plan --limit 30          # L4 Plan＝入力+スクショ（送信なし）
npm run serve                            # http://localhost:4599 で人間レビュー
```

スクショは `artifacts/plan_<companyId>.png`。承認ダッシュボードで 承認/却下/抑制 を操作。
